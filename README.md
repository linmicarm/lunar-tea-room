# Lunar Tea Room

A fictional tarot reader's portfolio site: a calm, earthy, "lightworker" space where a visitor can read about the reader (Selene), browse reading tiers, pull a free card from a working 78-card deck, and request a booking. Built as a vanilla HTML/CSS/JS project to practice design systems, token-driven theming, and building an interactive feature without a framework.

## What it does

- **Full-bleed hero** with a serif headline over a tarot photo and a deep-green wash for legibility.
- **About / Selene** section pairing a portrait with the reader's bio.
- **Readings ("Pick Your Pour")** — five tiers in a single hairline-divided row, tea-themed (One Sip, The Quick Sip, Two Cups, The Full Pot, The Year Ahead).
- **Free card pull** — the signature interactive feature. Tap the deck, it shuffles, and draws a random card from the full 78-card tarot deck with a name, suit emblem, keywords, and a plain-language meaning.
- **Testimonials** and a **booking form** (front-end only; no backend).

---

## Tech

Vanilla HTML, CSS (custom properties + Grid/Flexbox), and JavaScript. No build step, no dependencies. Fonts are Cormorant Garamond (serif display) and Jost (sans body) via Google Fonts. The deck data is generated by a small Python script (`gen_deck.py`) and written to `cards.js`.

```
index.html      # markup
style.css       # all styling, token-driven
main.js         # deck shuffle/draw + form + mobile nav
cards.js        # the 78-card deck data (generated)
gen_deck.py     # regenerates cards.js
assets/         # hero.jpg, reader.jpg
```

## Key decisions

| Decision | Why |
| --- | --- |
| Token-driven CSS (`:root` custom properties for color, type, spacing, structure) | One source of truth meant a full aesthetic overhaul could happen by editing variables instead of rewriting rules. |
| Kept legacy variable *names* as aliases during the reskin | Let the new earthy palette propagate across the whole sheet instantly; rules resolved to new values without touching every selector. |
| Generate the deck in Python, not hand-write 78 objects in JS | Less error-prone, and the meanings/keywords can be regenerated or tweaked in one place. |
| CSS-drawn suit emblems instead of emojis or icon files | No external assets, scales cleanly, and keeps the mystical tone consistent. |
| Hairline dividers + generous whitespace over cards/shadows | The calm "lightworker" aesthetic needed restraint; borders and color are used sparingly. |
| Photos hotlinked during design, downloaded locally for deploy | Faster iteration while choosing images; local files for a clean, dependency-free deploy. |

## Problems → Fix → Lesson

**Reading tier headings wrapped to two lines.**
Fix: widened the readings container (`max-width: 1320px`) and set `white-space: nowrap` on the headings.
Lesson: a 5-column row needs real horizontal room; constraining to the site's default max-width crushed the cells.

**The site drifted into "random colored tiles."**
An earlier version gave every section its own loud background color, so the page read as disconnected blocks instead of one design.
Fix: pulled back to a single shared base with color reserved for only a few key moments.
Lesson: cohesion comes from restraint and repetition, not from making every section a feature. (Hard-won — this was the second time the lesson came up.)

**Text became unreadable over patterned section backgrounds.**
Fix: removed background patterns from behind body copy; text sits on solid color only.
Lesson: decoration goes in empty zones, never behind paragraphs.

**Logo wordmark appeared twice (nav + hero lockup).**
Fix: removed the hero lockup; the nav carries the brand, the hero leads with the headline.
Lesson: repeating the wordmark within one viewport reads as redundant, not as branding.

**Em dashes scattered through the copy.**
Fix: rephrased every instance with periods, colons, or commas rather than swapping in another dash.
Lesson: removing a punctuation mark cleanly usually means rewriting the sentence, not find-and-replace.

## What I learned

- **A token system pays for itself.** The site went through several complete aesthetic pivots (kawaii sticker-book → bold color blocks → earthy mystical). Because color, type, and structure were all variables, each pivot was mostly an edit to `:root` plus targeted section fixes — not a rewrite.
- **Aliasing old variable names** is a fast, low-risk way to reskin a sheet that wasn't originally built for the new look.
- **Interactivity without a framework** is very doable: the card pull is plain DOM manipulation and a CSS class toggle for the shuffle animation.
- **Design is subtractive as often as additive.** Several rounds improved the site by *removing* things (patterns, extra colors, the duplicate logo, em dashes).
- **`white-space: nowrap` + container width** is the simple answer to "make these headings fit on one line."

---

## If I built it again / What I'd do differently

- **Start neutral, add color deliberately.** I'd begin with a calm base and introduce accents only where they earn attention, instead of color-blocking everything and walking it back.
- **Decide the brand voice and palette earlier.** A lot of iteration was discovering the aesthetic; a sharper brief up front would have saved pivots.
- **Build the card pull as a reusable component** with the data fully decoupled, so the same draw logic could power a multi-card spread (which the "Two Cups" / "Full Pot" tiers imply).
- **Add `prefers-reduced-motion` coverage** to the shuffle animation, and revisit focus states on the form for accessibility.
- **Lazy-load and size the hero image** more carefully for performance on slower connections.

## Running locally

No build step. Open `index.html` in a browser, or serve the folder:

```
python3 -m http.server
```

To regenerate the deck after editing card text:

```
python3 gen_deck.py
```
